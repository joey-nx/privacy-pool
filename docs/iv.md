# Invariants (불변 규칙)

> 프로젝트의 **절대적 강제 조항**을 정의한다.
> 모든 작업 시작 전 참고하고, 작업 완료 후 위반 여부를 검증해야 한다.

## MUST (반드시 해야 한다)

| # | 규칙 |
|---|------|
| M1 | 불확실할 때는 반드시 사용자에게 질문한다 |
| M2 | 새 코드 작성 전 기존 코드를 검색한다 (Glob/Grep) |
| M3 | 원자적(atomic) 단위로 작업하고 각 단계를 독립적으로 커밋 가능하게 유지한다 |
| M4 | 기존 코드베이스의 패턴과 구조를 따른다 |
| M5 | 코드 변경 전 영향 범위를 분석한다: ① 사용처 파악 ② 테스트 영향 ③ 동작 변경 여부 ④ 성능 영향 |
| M6 | 기존 로직에 영향이 있으면 사용자에게 먼저 확인한다 |
| M7 | 커밋 전 컴파일 에러와 경고가 없는지 확인한다 |
| M8 | 커밋 전 테스트 통과를 확인한다 (또는 테스트 변경에 대해 사용자 승인) |
| M9 | 3단계 이상 또는 아키텍처 결정이 필요한 작업은 플랜 모드로 진입한다 |
| M10 | 작업 전 이 문서를 참고하고, 작업 후 위반 여부를 자가 검증한다 |
| M11 | 비자명한 작업은 의도 문서(`specs/`)를 작성하고 사용자 승인 후 구현한다 |

## MUST NOT (절대 해서는 안 된다)

| # | 규칙 |
|---|------|
| N1 | 사용자 승인 없이 테스트 코드를 삭제/수정하지 않는다 |
| N2 | 중복 코드를 확인 없이 생성하지 않는다 |
| N3 | "나중에 필요할 수도 있는" 코드를 미리 추가하지 않는다 |
| N4 | 영향 분석 없이 기존 로직을 변경하지 않는다 |
| N5 | 프로덕션 코드에서 `.unwrap()`을 사용하지 않는다 |
| N6 | 컴파일러 경고를 무시하지 않는다 |
| N7 | 깨지거나 불완전한 코드를 커밋하지 않는다 |
| N8 | 관련 없는 변경을 하나의 커밋에 섞지 않는다 |
| N9 | 요구사항이 불명확한 상태로 작업을 진행하지 않는다 |
| N10 | 사이드 이펙트가 있는 변경을 사용자 확인 없이 진행하지 않는다 |
